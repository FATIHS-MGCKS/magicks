# SEO Audit — MAGICKS Studio

*Date: 2026-04-25 · Auditor: senior technical SEO + web quality engineer · Skill stack: `seo-audit` (jezweb-style framework) + `seo` (addyosmani / web-quality). Read alongside `.agents/product-marketing-context.md`.*

> **Scope**: full-site SEO inspection. **No code modified.** Findings only. Implementation order in §8.

---

## 1. Executive Summary

### Overall health: **B-**

The site is technically clean, fast, mobile-correct, and content-substantive. Information architecture is sane. NAP is consistent. Image alts are exceptional. Internal linking is well-structured. Title and description metadata is centralized in `src/app/seo/config.ts` and per-route `<RouteSEO />` injection works for client-side navigations.

But: **the site is a JavaScript-only React SPA** that injects per-route `<title>`, meta description, canonical, and OG tags at runtime via `useEffect`. The static `index.html` ships the homepage's title/description/canonical for **every** URL. Combined with **no `sitemap.xml` in the deploy artifact `public/`** (only emitted to `dist/` at build time, which is fine *if* `dist/` is what Hostinger serves) and **no JSON-LD schema anywhere**, the site is leaving the bulk of its rankable surface on the table.

### Top 5 priority issues

| # | Issue | Severity | Effort |
| --- | --- | --- | --- |
| 1 | SPA renders identical baseline meta/title/canonical for every route in initial HTML; only Google can render JS — Bing, social crawlers (LinkedIn, WhatsApp, X, Slack), and AI bots see homepage metadata everywhere. | **Critical** | High |
| 2 | No JSON-LD schema. No `Organization`, `LocalBusiness`, `WebSite`, `Service`, `BreadcrumbList`. Real Kassel address + email is unannotated. | **Critical** | Medium |
| 3 | OG image is **SVG** (`/og-default.svg`). Facebook, LinkedIn, WhatsApp, X reject SVG OG images; previews break on every share. | **Critical** | Low |
| 4 | `_redirects` is **Netlify/Cloudflare Pages format**; site is on **Hostinger** (Apache). The 301 from `/ueber-magicks` → `/ueber-uns` will not fire at the edge, and the SPA fallback rule is wrong format. | **High** | Low |
| 5 | Hero `<video>` has no `poster`; first paint is empty until CloudFront stream starts → poor LCP and no above-the-fold content for any crawler that doesn't autoplay. | **High** | Low |

### Quick wins (can be tackled in one short pass)

- Remove dead `applesocial.s3.amazonaws.com` `<link rel="preconnect">` from `index.html` — no resources are loaded from that origin anymore.
- Delete unreferenced `public/hero-video.mp4` (321.9 KB, only mentioned in a code comment).
- Convert `og-default.svg` → `og-default.png` (1200×630). Update reference in `seo/config.ts` and `index.html`.
- Replace `_redirects` with a Hostinger-correct `.htaccess`.

## 2. Critical Issues

### C-01 · SPA-only metadata (no SSR / no prerender)

- **Severity**: Critical
- **Affected**: every route, especially `/leistungen`, `/projekte`, all `/services/*` and `/seo/*` pages
- **Files**: `index.html`, `src/app/seo/SEO.tsx`, `src/app/seo/RouteSEO.tsx`, `vite.config.ts`
- **Problem**: `index.html` ships baseline meta — homepage title/description/canonical/OG tags. `<SEO />` runs in `useEffect` and updates `document.title` / `<meta>` *after* hydration. Initial HTML response is identical for `/`, `/leistungen`, `/webdesign-kassel`, `/projekte/renova-strassensanierung`, etc.
  - **Google**: renders JS during deferred indexing, but waits in render queue (rankings + freshness suffer).
  - **Bing**: limited JS rendering; will index every page with the homepage's title/description.
  - **Social crawlers**: Facebook, LinkedIn, WhatsApp, X, Slack, Telegram, Discord — none execute JS. Every share preview will be the homepage card, regardless of which URL was shared.
  - **AI bots**: GPTBot, ClaudeBot, Perplexity, etc. — all read static HTML; identical metadata everywhere.
- **Recommended fix**: Add a Vite prerender plugin that statically generates per-route HTML at build time (e.g. `vite-plugin-prerender-spa`, `vite-react-ssg`, or a small custom plugin that uses `react-dom/server` + the existing `SEO_BY_PATH` map to emit one `index.html` per static route into `dist/<route>/index.html`). React app continues to mount and hydrate as before. With a real prerender, every route ships its correct `<title>`, description, canonical, OG tags, and (with §C-02 fixed) JSON-LD before any JS runs.
- **Safe to fix automatically**: No — architectural change; needs build/test cycle and Hostinger SPA-rewrite review.
- **Expected impact**: Largest single SEO lift on the site. Unblocks Bing indexing, fixes all social previews, removes Google JS-render dependence, makes per-route descriptions actually appear in SERP snippets.

### C-02 · No JSON-LD schema markup anywhere

- **Severity**: Critical
- **Affected**: entire site
- **Evidence**: `rg "application/ld\+json"` returns **0 matches**. `index.html` has no `<script type="application/ld+json">`. `SEO.tsx` does not emit any structured data.
- **Problem**: Search engines can't read the studio's identity (Organization), local-business signals (address, area served, geo, opening hours), service catalog, breadcrumb trail, or per-page entity types. For a Kassel-local studio with a real address (Schwabstr. 7a, 34125 Kassel) and 14 indexable routes, this is a major missed signal.
- **Recommended fix**: Implement five schema types (in priority order):
  1. **`Organization` / `ProfessionalService`** — site-wide, includes `name`, `url`, `logo`, `email`, `address`, `sameAs` (when social profiles exist), `founder` (Fatih Serin), `foundingDate` (2024 — "EST. MMXXIV"), `slogan`.
  2. **`LocalBusiness`** subtype `ProfessionalService` — adds `geo` (51.319, 9.498 — Kassel center), `areaServed` (Kassel, Baunatal, Vellmar, Fuldabrück, Lohfelden, Niestetal, Kaufungen, Calden, Nordhessen, Deutschland), and `priceRange` if disclosed.
  3. **`WebSite`** — site-wide, includes `name`, `url`, `inLanguage: "de-DE"`, optional `potentialAction: SearchAction` if a site search is added later.
  4. **`Service`** — one per `/services/*` and `/seo/*` page (`Websites & Landing Pages`, `Shops & Produktkonfiguratoren`, `Web-Software`, `KI-Automationen & Integrationen`, `Website im Abo`, `Webdesign Kassel`, etc.) with `provider: { @id: Organization }` + `serviceType` + `areaServed`.
  5. **`BreadcrumbList`** — every non-homepage page (Home › Leistungen › Webdesign Kassel; Home › Projekte › Renova Straßensanierung).
- **Note**: `FAQPage` schema is **not** currently appropriate — the site has no visible FAQ blocks. Add only when a real FAQ section exists on a page.
- **Safe to fix automatically**: Yes for static schema; per-page Service schema needs the prerender from §C-01 to land in static HTML.
- **Expected impact**: Eligibility for Knowledge Panel, sitelinks, local pack, breadcrumbs in SERP, AI-Overview citations.

### C-03 · `og:image` is SVG (broken social previews)

- **Severity**: Critical (for social distribution)
- **Affected**: every URL, all share previews
- **Files**: `public/og-default.svg`, `index.html` (line 36, 47), `src/app/seo/config.ts` (line 18 `DEFAULT_OG_IMAGE`)
- **Problem**: Facebook/Meta, LinkedIn, WhatsApp, X (Twitter), Slack, Telegram and Discord do not render SVG as `og:image`. Most either show a broken/missing image or fall back to a domain-only stub. The current `og-default.svg` also references `Instrument Serif` and `SF Mono` fonts that won't be available to any preview renderer.
- **Recommended fix**: Generate a 1200×630 PNG (target file size 80–180 KB at quality ≈ 85) of the same composition. Save as `public/og-default.png`. Flip `DEFAULT_OG_IMAGE` to `${SITE_URL}/og-default.png` and update the baseline `og:image` / `twitter:image` in `index.html`. Per-page OG image overrides (already supported via `ogImage` in `SeoConfig`) can then ship per-route PNGs as proof points and case studies arrive.
- **Safe to fix automatically**: Yes — new asset + 2 string replacements. `sharp` is already a devDependency.
- **Expected impact**: Working share cards on every social platform; substantially better CTR on direct shares and DMs.

### C-04 · Wrong host redirect format for Hostinger

- **Severity**: Critical (silent — page-level still works, edge-level redirects don't)
- **Affected**: hosting layer (everywhere)
- **File**: `public/_redirects`
- **Problem**: `_redirects` is a Netlify / Cloudflare Pages syntax. Hostinger uses Apache → `.htaccess`. The legacy `/ueber-magicks → /ueber-uns 301!` rule **will never fire** at the edge; only the in-app `<Navigate>` (in `App.tsx` line 65) handles it client-side, after the SPA has mounted. Search engines see `/ueber-magicks` as a **soft 200** that flickers to `/ueber-uns`. The SPA fallback `/* → /index.html 200` is also unrecognized, meaning Hostinger's default 404 shows for any deep link refresh unless a separate `.htaccess` already handles it.
- **Recommended fix**: Replace `public/_redirects` with `public/.htaccess` containing:
  - `RewriteEngine On`
  - 301 redirect: `/ueber-magicks → /ueber-uns`
  - SPA fallback: rewrite all non-file/non-directory requests to `/index.html`
  - Optional: HTTPS enforcement, `Cache-Control` / `Expires` headers, `Strict-Transport-Security` (HSTS) — supports §C-01's prerender too.
- **Safe to fix automatically**: Yes — small file replacement, well-documented Apache rules.
- **Expected impact**: Working server-side 301 (preserves PageRank from any old `/ueber-magicks` backlinks), removed soft-404 risk, reliable SPA deep-link refreshes.

## 3. High-Priority Issues

### H-01 · Hero video has no `poster`; weak LCP + empty above-the-fold for crawlers

- **Severity**: High
- **File**: `src/app/components/HeroVideoBackground.tsx`, `src/app/heroMedia.ts`
- **Problem**: `<video>` autoplays, muted, looping, `preload="metadata"`, **no `poster` attribute**, source on CloudFront. Until first frame arrives, the entire hero is `bg-[#070708]`. The intro animation also pre-rolls a black wipe. Result: LCP candidate is the H1 text after JS executes (slow), and any crawler/preview that doesn't render JS sees a black hero.
- **Recommended fix**: Capture the hero video's first frame as a 1920×1080 WebP (≈ 60–120 KB), ship it at `/media/home/hero-poster.webp`, set `<video poster="/media/home/hero-poster.webp">`. Optionally also `<link rel="preload" as="image" href="/media/home/hero-poster.webp" fetchpriority="high">` in `index.html` to push it to LCP.
- **Safe to fix automatically**: Yes — asset + 1-line attribute + optional preload.
- **Expected impact**: LCP improvement (likely 0.3–0.7s on slow connections), better thumbnail in archive.org-style snapshots, no functional change to UX.

### H-02 · Homepage H1 carries no service keyword

- **Severity**: High (low cost, high upside)
- **File**: `src/app/components/home/Hero.tsx` (lines 21–22)
- **Problem**: H1 is *"Wir bauen das Web, das dein Business verdient."* — pure brand-poetic. While the `<title>` and meta description carry the service vocabulary, the H1 is the strongest on-page signal for the homepage's primary topic and currently includes none of: *Webdesign · Webentwicklung · Webagentur · Kassel · Studio*.
- **Recommended fix**: Two paths, both editorial-respectful:
  - **Option A — keep H1, add a short H2 below the lede**: e.g. *"Webagentur aus Kassel — für Websites, Shops, Web-Software & KI-Automationen"*. Visually treat as a quiet sub-line.
  - **Option B — modify H1 italic line**: e.g. *"Wir bauen das Web — / Webagentur aus Kassel."* Preserves brand cadence.
  - Recommend **Option A** to protect the Hero's restraint.
- **Safe to fix automatically**: No — copy decision; user-facing brand-line.
- **Expected impact**: Anchors the homepage to its primary brand + local + category cluster; helps for *"Webagentur Kassel"*, *"Webdesign Kassel"*, *"Webentwickler Kassel"* SERPs.

### H-03 · Several `<title>` tags too short — wastes SERP real estate

- **Severity**: High
- **File**: `src/app/seo/config.ts`
- **Problem**: Title tags should land 50–60 characters with primary keyword + brand. Several pages are too short to compete:
  | Path | Current title | Length | Issue |
  | --- | --- | --- | --- |
  | `/ueber-uns` | "Über uns – MAGICKS Studio" | 25 | misses keywords ("kreatives Tech-Studio", "Webagentur Kassel") |
  | `/kontakt` | "Kontakt – MAGICKS Studio" | 24 | misses CTA-like value (e.g. "anfragen", "unverbindlich") |
  | `/web-software` | "Web-Software – MAGICKS Studio" | 29 | misses *individuelle*, *Dashboards*, *für Unternehmen* |
  | `/website-im-abo` | "Website im Abo – MAGICKS Studio" | 32 | misses *monatlich*, *individuelle*, *Kassel* |
  | `/webdesign-kassel` | "Webdesign Kassel – MAGICKS Studio" | 34 | could expand to include *Webagentur* / *für Unternehmen* |
  | `/landingpages-kassel` | "Landing Pages Kassel – MAGICKS Studio" | 37 | could include *erstellen lassen* / *Conversion* |

  Counter-issue: `/leistungen` ("Leistungen von MAGICKS Studio – Websites, Shops, Software & Automationen") is 74 chars — Google will truncate.
- **Recommended fix**: Rewrite titles to land 52–58 chars with primary keyword forward + brand at end. Examples (proposals, copy decisions live in §6):
  - `/ueber-uns` → "Über uns — Kreatives Tech-Studio aus Kassel | MAGICKS Studio" (60)
  - `/kontakt` → "Projekt anfragen — Kontakt zu MAGICKS Studio aus Kassel" (55)
  - `/web-software` → "Web-Software & Dashboards für Unternehmen | MAGICKS Studio" (58)
  - `/webdesign-kassel` → "Webdesign Kassel — Webagentur für Unternehmen | MAGICKS" (54)
  - `/leistungen` → "Leistungen — Websites, Shops, Software, KI | MAGICKS Studio" (60)
- **Safe to fix automatically**: No — copy decisions; should be coordinated with the upcoming `copywriting` skill pass.
- **Expected impact**: Better SERP CTR + keyword alignment for short-titled routes.

### H-04 · Sitemap not committed to repo / not in `public/`

- **Severity**: High (only matters if Hostinger deploys from a stale build, but likely)
- **File**: `vite.config.ts` (sitemap plugin emits to `dist/sitemap.xml` only at build time), `public/robots.txt` (line 9 references `https://magicks.studio/sitemap.xml`)
- **Problem**: `sitemap.xml` exists only as a build artifact. If a deploy happens without a fresh `npm run build`, or Hostinger serves `public/` directly, `magicks.studio/sitemap.xml` returns 404. `robots.txt` advertises a path that may not exist. Verified: `dist/` directory is not present in the working tree, and there's no committed sitemap.
- **Recommended fix**: Two acceptable paths:
  - **(a) Verify deploy pipeline**: confirm Hostinger receives `dist/` (from a CI build or local `npm run build` upload). If yes, document this clearly in a `DEPLOY.md` to prevent regression.
  - **(b) Commit a static `public/sitemap.xml`**: regenerate it once via the existing plugin logic and check it in. Lighter pipeline, but loses the build-time auto-detection of new featured projects — would need a manual update step.
- **Recommend (a)** — keep the build-time generator, ensure deploy uploads `dist/`.
- **Safe to fix automatically**: Investigation step (no code change).
- **Expected impact**: Removes 404 risk on the canonical sitemap URL. Avoids GSC submission errors.

### H-05 · `applesocial.s3.amazonaws.com` preconnect is dead weight

- **Recommended fix**: Delete the line. Keep the `cloudfront.net` preconnect (still used by hero video) and Google Fonts preconnects.
- **Safe to fix automatically**: Yes — single-line removal.
- **Expected impact**: Marginal load-time improvement; cleaner DevTools console.

### H-06 · Editorial section in DatenschutzPage `§ 05` heading mentions removed AWS S3 SF Pro
